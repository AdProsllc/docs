The Subscription Intelligence Layer

A unified agent framework for delivering proactive, opinionated, daily intelligence to Ad Pros clients — built on one engine, three subscription tiers, eight digest types.

8 agents 3 tiers 6 weeks to launch ~95% gross margin
01.01—The Thesis

Most agencies hide behind monthly reports. The agency that shows up in a client's inbox every morning at 8am — with sharp, opinionated, data-backed intelligence — is perceived as 10x more valuable, regardless of actual hours worked.

Core Insight

We are not building seven agents. We are building one agent framework with seven subscription types. The product is the subscription model itself — a daily, chat-extensible intelligence layer that sits on top of our existing data infrastructure.

Every agent in the framework follows the same primitive:

03.01—Data Schema

Five tables. Postgres on Supabase. The schema is deliberately minimal — every agent type fits into the same five-table model via the filter_payload jsonb and agent_targets join.

Why JSONB for Filters

Each agent type has different config needs (a competitor agent stores domains, a campaign agent stores Meta campaign IDs, a concept agent stores concept tags). Rather than 8 different tables, we use one filter_payload jsonb column with per-agent schema validation in the application layer. This keeps adding new agents to a single migration.

04.01—The Five-Phase Lifecycle

Every agent — regardless of type — follows the same five phases. This is the framework's contract. New agent types only have to implement two things: a data adapter and a prompt template.

01
Trigger

Scheduler fires

Cron job (every 15 min) reads subscriptions where next_run_at ≤ now AND status = active. Each match becomes a job in the digest queue (BullMQ on Redis).

INsubscriptions table→OUTdigest_jobs queue
02
Gather

Pull metrics

Worker picks up job → calls the agent-type-specific data adapter (BigQuery, Meta API, AdScan, scraper) → returns a normalized metrics_snapshot jsonb.

INsubscription + filter_payload→OUTmetrics_snapshot jsonb
03
Compose

Claude writes the digest

Snapshot + agent prompt template + client KPI definitions are sent to Claude. Output is structured markdown: headline → 5-10 metric rows → 1-3 opinionated observations → "chat for details" CTA.

INmetrics_snapshot + prompt→OUTdigest_text + digest_html
04
Deliver

Multi-channel send

Email via SendGrid → Slack via webhook → in-app inbox row. Each delivery includes a chat handoff URL with a pre-generated session token (?chat=<uuid>).

INdigest_text + recipient list→OUTdelivery_status logged
05
Extend

Chat for details

Client clicks the URL → chat session is hydrated with metrics_snapshot + digest_text as system context → client can ask any follow-up. The agent now feels alive.

INsession_token→OUTlive conversation
04.02—The Digest Prompt Template

Every agent uses a variant of this base prompt. The variable bits are {agent_role}, {metrics_snapshot}, and {client_context}.

SYSTEM:
You are an Ad Pros analyst writing a daily intelligence brief for a DTC
ecommerce client. Your tone is sharp, opinionated, and confident — like
a senior strategist who has been watching their account all night.

You write in the style of a Bloomberg terminal alert: lead with the
headline, then numbers, then a take. Never hedge. Never use words like
"may" or "potentially." If something is broken, say it's broken.

ROLE:
{agent_role}  // e.g. "You are monitoring the 'UGC-testimonial' concept"

CLIENT CONTEXT:
- Brand: {client.brand_name}
- Industry: {client.vertical}
- Active KPIs and goals: {client.kpi_definitions}
- Last 7 days baseline: {client.baseline}

TODAY'S DATA:
{metrics_snapshot}

OUTPUT FORMAT (markdown):
1. Headline (one line, present tense, opinionated)
2. Metrics table (5-10 rows: metric · today · vs goal · vs 7d avg · status)
3. What's working (1-2 bullets, max)
4. What's broken (1-2 bullets, max)
5. What we're doing about it (1-2 bullets, max)
6. CTA: "Chat for details →"

CONSTRAINTS:
- Max 180 words total
- Numbers must be exact, never rounded misleadingly
- If a metric is missing, write "no data" — never invent
- Never use emoji except ▲ ▼ ─ for trend indicators
04.03—Adding a New Agent Type

Once the framework is live, adding a new agent (e.g., "creative fatigue digest") is a four-file change:

adapters/
  └── creative_fatigue.ts    // pulls data → returns metrics_snapshot

prompts/
  └── creative_fatigue.md    // the {agent_role} text

migrations/
  └── 2026_xx_add_creative_fatigue_enum.sql

ui/
  └── subscribe/creative_fatigue.tsx   // the opt-in form

Everything else — scheduler, formatter, delivery, chat handoff — is reused. Build time for a new agent: ~1 day after the framework exists.
