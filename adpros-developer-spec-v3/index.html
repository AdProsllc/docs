01 · Executive Summary

One decision drives everything: pre-compute what can be pre-computed, pay for what must be live.

At 1,000+ brands and 100K new ads/month, the cost problem isn't ingestion — it's synthesis. The naive path for "analyze this brand" is 1.6M tokens per query and context-rot failure. Solution: three-layer architecture + hot/warm/cold brand tiering + Haiku-first cascade routing + disciplined caching. This document is the build spec.

The Core Insight
This isn't a context-window problem. It's a context-hierarchy problem. The window is never big enough; the right question is which tier of pre-computed summary to consult, and when to escalate to raw data. Get the hierarchy right and token cost collapses without quality loss.

The Three-Layer Architecture

L1 — Atomic. Per-asset structured JSON against 12-dim rubric. Haiku 4.5 via Batch API. Shared cross-tenant. ~$1.20 per 1K ads.

L2 — Brand Digests. Six pre-computed rollups: Pattern Library, Hook Taxonomy, Messaging Index, LP Graph, Performance Proxy, Strategy Brief. 30–60K tokens total regardless of ad count.

L3 — Synthesis Agent. Claude answers via four tools in cost order: query_brand_digest → search_ads → get_ad_analysis → inspect_media. Context editing at 40% fill. Memory tool for session persistence.

What Changed From v2 Based on Latest Research

Explicit 1h TTL on all caches. Anthropic silently defaulted to 5m on March 6, 2026. Every cache_control needs explicit ttl: "3600".

Cascade replaces classifier. Haiku-first with confidence-scored escalation. 51% cost reduction vs uniform Opus per Augment Code. Classifier pre-selection is brittle.

Eviction threshold at 40%, not 80%. Context rot kicks in at 30–40% window fill per Chroma. Evict early, not late.

CI enforcement of prefix stability. Byte-compare test on every commit. Cache invalidation is the #1 silent cost killer.

Hot / Warm / Cold Brand Tiering

Hot (queried last 7d · ~50–100 brands): nightly refresh, pre-warmed cache, all six digests.

Warm (queried last 30d · ~150–300 brands): weekly refresh, incremental updates, Strategy Brief weekly only.

Cold (30d+ dormant · ~600–800 brands): no pre-built digest. First query triggers sync build with progress narration UX, then promotes to Warm.

→ 85-90% digest cost reduction vs refresh-all-nightly.

What this is explicitly not

Rejected Approaches
  • Not vector RAG over raw ad text. Over-retrieves, kills cache, produces context-rot failure (Chroma 2025, 18 models). Digests are the retrieval layer.
  • Not a 1M-context play. NoLiMa: 10 of 12 models with ≥128K claimed drop to ≤50% of base by 32K. Effective usable context is 10–25% of claimed.
  • Not fine-tuning per brand. Claude + well-engineered context beats fine-tuned small models on analysis. Retraining infeasible at 1,000+ brands.
  • Not multi-agent by default. Cognition: single-threaded with compaction wins for sequential work. Multi-agent only for proven parallel read-only tasks.
  • Not classifier-based tier routing. Cascade with confidence escalation is robust; classifier pre-selection is brittle and pays full tier cost on misprediction.

The three-layer hierarchy

L1
Atomic

Per-asset structured analysis

Every image, video (frame set + transcript), and landing page gets analyzed once and stored as strict JSON against the 12-Dimension rubric. No prose. Tiered vision routing: ~80% Haiku 4.5, ~15% Sonnet 4.6, ~5% Opus 4.7 for calibration.

Critical at SaaS scale: Keyed on ad_external_id, not tenant ID. Shared across tenants. If tenants A and B both watch Nike, compute once; amortize cost across requesters over 30-day window.

Storage: Supabase ad_visual_analysis, ad_video_analysis, ad_landing_pages (tenant-agnostic) · Compute: Haiku 4.5 via Batch API with seed-then-fanout · Cost: ~$1.20/1K ads with 95% cache read rate · Schema version: Every row tagged with rubric_version
L2
Digests

Pre-computed brand digests (six artifacts per brand)

Collapses thousands of atomic analyses into 30–60K tokens regardless of ad count. Artifacts: Pattern Library (8–15K), Hook Taxonomy (3–5K), Messaging Index (5–10K), LP Graph (10–20K), Performance Proxy (4–8K), Strategy Brief (~2K). Hard-capped at write time.

Refresh cadence by brand tier: Hot = nightly. Warm = weekly incremental. Cold = on-demand build at first query. Digest token budget enforced on every write — if output exceeds cap, auto-compact pass.

Storage: Supabase brand_digests (JSONB) + R2 markdown mirror · Compute: Sonnet 4.6 via Batch with seed-then-fanout · Cost: Hot $0.15/brand/day, Warm ~$0.02/brand/day, Cold $0 dormant · Eval: Regression suite runs pre-promotion; >5% quality drop rejects new digest
L3
Agent

Synthesis agent with Haiku-first cascade

Four tools in strict cost order: query_brand_digest(sections[]) → search_ads(filters, limit) → get_ad_analysis(ids[]) → inspect_media(id). Cascade tries Haiku first; escalates to Sonnet on low confidence (<0.7) or schema parse failure; Opus only on explicit validator flag.

Context management primitives: clear_tool_uses_20250919 triggered at 40% window fill (below 30–40% context rot zone), with exclude_tools: ["memory"]. Memory tool (memory_20250818) for cross-session persistence. Per-tenant cache isolation via workspace-level keying.

Primary model: Sonnet 4.6 for standard queries · Cascade: Haiku 4.5 → Sonnet 4.6 → Opus 4.7 with confidence scoring · Cache breakpoints: 3 — system+tools (1h TTL), brand digest (1h TTL), conversation (context-editing)

Silent cache killers (must be prevented)
  • Timestamps, user IDs, request IDs, trace IDs in any cached block — move to dynamic suffix
  • Swift/Go JSON key randomization — enforce sort_keys=True, byte-compare in CI
  • Framework upgrades reordering tool defs — pin versions, snapshot tests on upgrades
  • Toggling web_search, citations, tool_choice between requests — segment cache families
  • Prompts under 4,096 tokens for Haiku 4.5 — pad stable rubric content
  • Default 5m TTL (Anthropic March 6, 2026 silent change) — explicit "ttl": "3600" on every breakpoint
  • Dynamic content inserted before stable content — every cached block must come before the first dynamic token

Hot / Warm / Cold brand tiering

Hot tier

Definition: Queried in last 7 days, or 3+ queries in 7d, or Enterprise-tier tenant watchlist.

Refresh: Nightly full rebuild via Batch API (seed-then-fanout).

Cache: Pre-warmed 6am local time.

Cost: ~$0.15/brand/day.

Promotion: Warm → Hot on 3+ queries in 7d or subagent audit request.

Warm tier

Definition: Queried in last 30 days but not currently Hot.

Refresh: Sunday full rebuild + incremental updates mid-week.

Cache: Write on first query each day.

Cost: ~$0.02/brand/day.

Strategy Brief: Weekly only (not nightly — too expensive).

Cold tier

Definition: No queries in 30+ days. Digest artifacts preserved in R2 cold storage for 90 days, then purged.

Refresh: None while dormant.

Cache: Built on first query (synchronous, ~30-60s UX with progress narration).

Cost: $0 dormant.

Promotion: Cold → Warm on first query.

Cascade routing (not classifier)

Query arrives │ ├─ Haiku 4.5 attempt │ structured output w/ confidence field │ ├─ confidence >= 0.7 && valid schema → DONE │ └─ else → escalate with Haiku attempt as context │ ├─ Sonnet 4.6 escalation │ structured output w/ confidence field │ ├─ confidence >= 0.8 && valid schema → DONE │ └─ else → escalate if validator flags │ └─ Opus 4.7 (rare, <5% of queries) explicit deep-reasoning path used for: premium audits, hardest edge cases // Expected distribution: 60% Haiku · 35% Sonnet · 5% Opus // Augment Code published: 51% cost cut vs uniform Opus 4.6
Why cascade, not classifier
A classifier has to be right before any real tokens are spent. Mispredictions cost full-tier price on every wrong route. A cascade tries cheapest first and escalates only on measurable failure. Same distribution, but misroutes cost ~Haiku price + ~Sonnet price, not 2× Sonnet.

Haiku-first cascade (not classifier)P05 days51% cost cut
Cascade: Haiku 4.5 attempts first. Escalate to Sonnet 4.6 only on low confidence (<0.7) or schema parse failure. Opus 4.7 only on explicit validator flag. Augment Code: 51% cost reduction vs uniform Opus 4.6.
▶
Sub-tasks
  • Structured output schema includes confidence: float field for all atomic/synthesis calls
  • Implement CascadeRouter: try Haiku → escalate on (low confidence | schema fail | explicit flag)
  • Pass Haiku attempt as context to Sonnet on escalation (not fresh retry)
  • Log cascade path per request: haiku_only | escalated_to_sonnet | escalated_to_opus
  • Confidence thresholds tunable via config (start: Haiku 0.7, Sonnet 0.8)
Code pattern
async def cascade_route(query, context): haiku = await call_haiku(query, context, schema=ResponseSchema) if haiku.confidence >= 0.7 and haiku.schema_valid: log_cascade("haiku_only") return haiku sonnet = await call_sonnet(query, context, schema=ResponseSchema, prior_attempt=haiku) # agent sees what Haiku tried if sonnet.confidence >= 0.8 and sonnet.schema_valid: log_cascade("escalated_to_sonnet") return sonnet if requires_deep_reasoning(query, sonnet): log_cascade("escalated_to_opus") return await call_opus(query, context, prior_attempts=[haiku, sonnet]) return sonnet
Acceptance criteria
DONE WHEN: (1) Cascade distribution: 60% Haiku-only / 35% escalated-to-Sonnet / 5% escalated-to-Opus; (2) Cost blended per query ≤$0.04; (3) Quality eval pass rate within 2 points of uniform-Opus baseline; (4) Misroute cost measurable and <20% of total spend.
Pitfall
WATCH: Don't retry from scratch on escalation. Pass the Haiku/Sonnet attempt as context so the higher tier builds on prior work. Fresh retry wastes tokens.
