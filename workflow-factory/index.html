The Core Insight

The factory is itself an instance of the 8-step flow — and it's self-directing.

Every workflow Ad Pros produces follows Collect → Organize → Remember → Think → Plan → Do → Review → Improve. The factory is no different. But the real breakthrough isn't just that the factory builds workflows — it's that the factory decides which workflows to build next.

When existing workflows run, the Improve Agent sees where humans kept editing briefs, which clients kept asking for new analyses, which gaps the current library can't fill. It feeds those signals to the Recommender Agent, which proposes the next workflow to build. Humans approve the queue — they don't invent it. This is what makes it autonomous: the factory's backlog is written by the factory itself.

Every workflow shipped teaches the factory to ship the next one better and to know what to ship next. That compounding loop is the whole bet.

06 · Research Foundation

Not invention. Composition.

Every piece of this factory maps to a published research paper, production system, or engineering playbook from 2022–2026. This tab walks through the actual logic, the papers, the GitHub repos, and why each one earns its place in the architecture.

The mental model: outer loop searches; inner loop optimizes; skills persist; evals gate.

Every serious agent-factory in the research literature follows the same four-tier mental model. (1) An outer loop that searches over workflow topologies — what stages should exist, what agents should run in parallel, where should routing happen. (2) An inner loop that optimizes prompts and weights within a chosen topology. (3) A persistent skill library that grows over time, so workflow 100 can reuse what workflow 47 learned. (4) An eval harness that gates everything — topology choices, prompt changes, deployment decisions.

Miss any one of the four and the factory collapses: no outer loop means you hand-author topologies, no inner loop means you hand-tune prompts, no skill library means every workflow starts from zero, no evals means you ship 100 unmeasured liabilities. The research consensus (Khattab/DSPy, Zhang/MetaGPT, Hu/ADAS, Husain/evals, Anthropic/Skills) is that all four must be built together — and the rest of this tab maps that consensus to the concrete agents in the Ad Pros factory.

A · The Four Core Frameworks

What each factory agent actually runs on.

These four papers/systems are the load-bearing research the factory is built on. Each has a working implementation, measured performance, and a direct mapping to one or more Ad Pros factory agents.

AFlow (MetaGPT)
ICLR 2025 Oral · Zhang et al.
Outer Loop · Topology Search
Represents agentic workflows as code-based graphs of Operators and Nodes. Uses Monte Carlo Tree Search (MCTS) to search over possible workflow topologies, given only an eval function. Generates workflows in ~1.5 hours that beat hand-designed workflows by 5.7% average across Coding, Math, QA. Matches GPT-4o performance at 4.55% of its cost on HumanEval.
How the factory uses it
The Shape Agent and Stage Designers use AFlow-style topology reasoning when they compose a new workflow. Instead of running full MCTS per request (too slow), the factory pre-searches MCTS across the 8 canonical shapes offline, then the Shape Agent selects from those cached topologies. Cold-path MCTS. Hot-path retrieval.
arxiv.org/abs/2410.10762 github.com/FoundationAgents/AFlow
DSPy + GEPA
Stanford · Khattab · 2023–2025
Inner Loop · Prompt Optimization
DSPy treats LM pipelines as programs, not prompts — declarative Modules with typed Signatures, compiled by outer-loop optimizers. GEPA (Jul 2025) is the state-of-the-art optimizer: reflective prompt evolution, outperforms reinforcement learning methods using 35× fewer rollouts. Think: genetic algorithm where each mutation is a reflection-driven prompt edit.
How the factory uses it
The Improve Agent runs GEPA-style reflection on each deployed workflow. When human edits to a Stage Designer's output show a pattern (e.g., humans keep adding Pydantic schemas to Stage 4), GEPA proposes a prompt patch: "add explicit Pydantic output schema requirement to Stage 4 template." The patch is PR'd to the factory's prompt repo. Workflow 11+ gets the improved prompt on first try.
DSPy: arxiv.org/abs/2310.03714 GEPA: arxiv.org/abs/2507.19457 github.com/stanfordnlp/dspy
ADAS (Meta Agent Search)
ICLR 2025 · Hu/Lu/Clune · UBC
Meta-Loop · Agent Discovery
A meta-agent iteratively programs new agents in code, tests them on held-out tasks, adds successful ones to an archive, and uses the archive to inform further discoveries. Discovers novel agent architectures that outperform human-designed baselines. The purest published instance of "agent-designs-agents."
How the factory uses it
The Recommender Agent runs an ADAS-style search over the workflow backlog, not agent code. It proposes candidate workflows by recombining successful patterns from the archive (past deployed workflows), scores them against signal gaps, and ranks. The Shape Library is the archive. New shapes get born when ADAS-style search finds gap-filling combinations.
arxiv.org/abs/2408.08435 github.com/ShengranHu/ADAS
Claude Agent Skills
Anthropic · Oct 2025 · open Dec 2025
Persistence · Skill Library
Progressive disclosure: name + description always loaded (~50 tokens), SKILL.md body loaded on trigger, scripts/references loaded on demand. Solves the "bloated system prompt" problem directly. Open standard since Dec 2025 — adopted by Microsoft, Cursor, Goose, Amp; partners include Atlassian, Canva, Figma, Notion, Stripe, Zapier.
How the factory uses it
The Skills Curator outputs every new Skill as a SKILL.md file following this standard. The LTV template's three Skills (supplement-economics.md, metric-definitions.md, compliance.md) are exactly this format. Factory-generated workflows can load any Skill from the Library at any stage without context bloat, because only the 50-token header is ever in the system prompt — the body loads just-in-time when the stage references it.
anthropic.com/engineering/agent-skills github.com/anthropics/skills
B · The Five Canonical Patterns

The vocabulary the Stage Designers emit.

From Anthropic's "Building Effective Agents" (Dec 2024, Schluntz & Zhang). Every serious framework (LangGraph, CrewAI, AutoGen, Semantic Kernel) has converged on these five primitives plus the augmented LLM base. This is the only pattern vocabulary the Stage Designers are allowed to emit — banning exotic patterns is what keeps the factory's output auditable.

01

Prompt Chaining

Decompose a task into a fixed sequence of LLM calls. Each step processes the previous step's output. Add programmatic checks ("gates") between calls.

Used in LTV stages: 0.1, 0.2, 1.1–1.4, 3.1, 9.x
02

Routing

Classify input and direct to a specialized follow-up. Different prompts/models for different paths. Enables separation of concerns.

Used in LTV stage: 4.1 (LTV Method Router — picks Bayesian / BG-NBD / Contractual / Triangle)
03

Parallelization

Run multiple LLM calls simultaneously then aggregate. Two flavors: sectioning (split into independent subtasks) and voting (multiple runs for consensus).

Used in LTV stages: 2.1, 2.2, 2.3 (cohort analyses); 7.1–7.7 (brief writers)
04

Orchestrator-Workers

A central LLM dynamically breaks down tasks, delegates to worker LLMs, synthesizes. Workers don't know about each other. Essential for unbounded/dynamic work.

Used in LTV stage: 5.1–5.3 (Per-SKU Worker Spawn — one worker per SKU, orchestrator joins)
05

Evaluator-Optimizer

Generator LLM produces output, evaluator LLM critiques, loop until acceptance criteria met. Works when LLMs can critique better than they can one-shot.

Used in LTV stage: 3.2 (Numbers Reviewer — loops up to 3 times on failure, then escalates)
06

+ Augmented LLM (base)

The foundation: LLM + retrieval + tools + memory. Every other pattern composes over this. Not a separate pattern — the substrate everything runs on.

Used in LTV: every Autopilot Agent + Super Agent runs as augmented LLM
C · The Architecture Debate

Cognition vs. Anthropic, and why it matters every time the factory runs.

June 2025 saw the most important architectural argument in applied agents: multi-agent vs. single-context. The factory must take a position on this for every workflow it generates. Here's the debate, the resolution, and how the Shape Agent encodes it.

Cognition: "Don't Build Multi-Agents"
Walden Yan · June 12, 2025
Single-Context Camp
Core claim: parallel subagents make conflicting implicit decisions. The canonical example: one subagent generates a Mario-style background, another generates a Flappy-Bird-style bird, neither saw the other's implicit choices, result is incoherent. Solution: one agent, one context, shared state, linear execution.
cognition.ai/blog/dont-build-multi-agents
Anthropic: Multi-Agent Research System
June 13, 2025 · next-day response
Multi-Agent Camp
LeadResearcher (Opus) plans → spawns parallel Sonnet subagents with separate contexts → synthesizes. 90.2% improvement over single-agent Opus on internal research evals, at ~15× the tokens. Explicit task boundaries + orchestrator persistence + rainbow deployment handle the coordination problem.
anthropic.com/engineering/multi-agent-research-system
Chase Reconciliation
LangChain · June 16, 2025
The Resolution
Both sides are right, for different workload types: read-heavy tasks (research, analysis, retrieval) benefit from multi-agent because subagents return synthesized outputs, not conflicting artifacts. Write-heavy tasks (code, coordinated content) stay single-context because one coherent decision-maker beats parallel implicit decisions. Tag every workflow accordingly.
blog.langchain.com/how-and-when-to-build-multi-agent-systems
How the Factory Encodes It
Ad Pros · architectural rule
Binary Tag · Per Workflow
The Shape Agent tags every workflow as read-heavy or write-heavy. Read-heavy (LTV, creative fatigue, attribution analysis) → Stage Designers are allowed to use Orchestrator-Workers + Parallelization. Write-heavy (campaign restructure, budget reallocation, PDP copy changes) → single-context only; parallelization banned; Evaluator-Optimizer loops instead. This tag is non-negotiable and the Eval Agent enforces it.
D · Self-Evolution Mechanics

How the factory learns from itself.

Three systems establish the mental model for agents that write, store, retrieve, and improve their own skills over time. Without these, the factory is a one-shot generator — with them, workflow 100 is materially better than workflow 1.

Voyager
NVIDIA · Wang et al. · 2023
Skill Library Growth
LLM-powered Minecraft agent with three components: (1) automatic curriculum that proposes progressively harder goals, (2) skill library of executable code for behaviors, (3) iterative prompting with environment feedback, self-verification, and code refinement. The agent writes its own skills, stores them, retrieves them for composition.
How the factory uses it
The Skills Curator + Improve Agent together implement Voyager's three components. Curriculum = the Recommender's backlog. Skill library = the SKILL.md repo. Iterative refinement = the Improve Agent's PR-based prompt edits after observing real workflow runs.
arxiv.org/abs/2305.16291 github.com/MineDojo/Voyager
Reflexion
Shinn et al. · 2023
Self-Critique Loop
Agent receives feedback signal, converts it to verbal reflection, stores the reflection in episodic memory, uses it as additional context for the next trial. No weight updates — pure prompting with reflection traces. Simple, cheap, and foundational for every self-improving loop since.
How the factory uses it
The Improve Agent runs Reflexion loops on failed workflow runs (evals failed, human overrode output, cost cap hit). Each failure becomes a verbal reflection stored against the relevant factory agent. Those reflections enter the next-run system prompt as few-shot examples.
arxiv.org/abs/2303.11366
BabyAGI 2o
Yohei Nakajima · 2024
Self-Tooling
Agent dynamically creates and registers new Python tools at runtime. Hits a capability it lacks → writes a tool for it → adds it to the registry → uses it. Function-registry stored with embedding search, triggers, authed secrets, and logs. The purest public instance of "agent-builds-tool."
How the factory uses it
Informs the Recommender's autonomy model: when the Recommender identifies a gap that needs a new Skill, it doesn't just propose the workflow — it proposes the Skill that would need to exist to power the workflow. Skills Curator drafts. Human approves. Registry grows.
github.com/yoheinakajima/babyagi-2o
E · Eval Discipline

Why the Eval Agent is the most important factory agent.

Three 2025 voices converge on the same point: evaluation is the bottleneck, not model capability. A factory shipping agents without evals is shipping tech debt at 100× the rate. Here's the consensus and how the factory implements it.

Hamel Husain (Parlance Labs)
A Field Guide to Rapidly Improving AI Products
The Error-Analysis Doctrine
Start with error analysis on 20–50 real traces, not eval-driven dogma. Use a domain-expert "benevolent dictator" as evaluator. Avoid generic criteria ("is it helpful?"). Build evals from observed failures, not imagined ones.
How the factory uses it
The Eval Agent doesn't generate evals from theory. It reads the Improve Agent's failure archive (real traces from past workflows) and clones the test patterns that caught those failures into the new workflow's eval harness.
hamel.dev/blog/posts/evals-faq
Shunyu Yao (Princeton/OpenAI)
"The Second Half" · April 2025
The Evaluation-Bottleneck Thesis
Reframes AI progress as evaluation-limited, not capability-limited. The "second half" of AI progress is not better models — it's better benchmarks, better evals, better scoring. Whoever ships the best evals ships the best agents.
How the factory uses it
Eval Agent has first-class status equal to Stage Designers, not subordinate. Evals ship alongside the workflow; they are the deliverable, not the afterthought.
ysymyth.github.io/The-Second-Half
Eugene Yan
Patterns for Building LLM-based Systems
The Operational Checklist
Canonical pattern list: evals, retrieval augmented generation, fine-tuning, caching, guardrails, defensive UX, collect user feedback. The closest thing applied LLM engineering has to a standard checklist.
How the factory uses it
Every Stage Designer's output must address each of the seven patterns or explicitly decline with rationale. The Eval Agent validates this as part of the Blueprint check.
eugeneyan.com/writing/llm-patterns
Braintrust + Langfuse
Production eval tooling · 2024–26

Production tools for running evals continuously against deployed agents. Traces + scorers + dashboards + regression detection. What the factory deploys its Eval Agent output to.

F · Failure Modes the Factory Must Plan For

What goes wrong, and how each failure is prevented.

Six failure modes come up repeatedly across credible critiques of agent factories. Each has a documented origin, an engineering mitigation, and a factory-agent that enforces the mitigation.

Failure 01

Autonomy-Hallucination Gap

Karpathy (Oct 2025): reliability improves exponentially slower than capability. "Decade of agents, not year of agents." The Waymo analogy: 10 years from demo to limited deployment.

Mitigation: 3 human gates + eval-gated deploy. Factory never ships an unevaluated workflow.
Failure 02

Context Rot

Anthropic (Sept 2025): performance degradation as context lengthens. Multi-agent systems burn ~15× more tokens than single-agent.

Mitigation: Claude Skills progressive disclosure + compaction + structured note-taking + sub-context isolation per stage.
Failure 03

Conflicting Implicit Decisions

Cognition (June 2025): parallel subagents make implicit choices that don't align. Mario background + Flappy Bird bird.

Mitigation: Shape Agent tags read-heavy vs write-heavy; write-heavy workflows are banned from using parallel patterns.
Failure 04

Eval Debt

Hamel Husain: teams ship LLM products without evals, accumulate unmeasured failures, hit a wall where no one knows why performance regressed.

Mitigation: Eval Agent runs before Deploy. No eval = no deploy. Regression monitoring via Braintrust/Langfuse.
Failure 05

Lethal Trifecta

Simon Willison: private data access + untrusted input + external communication = exfiltration via prompt injection. Existential for any factory with ClickUp write access.

Mitigation: Deploy Agent has scoped MCP tokens, idempotency guards, audit logs, treat-user-input-as-hostile on any externally-triggered workflow.
Failure 06

Demoware-to-Production Gap

METR (July 2025): experienced devs were 19% slower with AI tools despite feeling 20% faster. Production agents need retries, checkpoints, idempotency, circuit breakers.

Mitigation: Every Stage Designer's output must include cost caps, retry logic, idempotency keys, and a circuit breaker. Non-negotiable Blueprint requirements.

The Custom-Field Constraint
API reality · verified
Deploy Agent · Must Work Around
Custom field definitions cannot be created via API — only read/written. This is the single biggest architectural constraint. Naive factory: fails. Ad Pros factory: pre-builds a canonical Field Template Pack once, reuses across all 100 workflows.
How the factory handles it
The LTV template's 22 custom fields become the Field Template Pack. Deploy Agent never tries to create fields — it inherits them from the workspace's canonical schema and only sets values on tasks. The Blueprint must declare which Pack fields it needs; the Eval Agent rejects Blueprints requiring fields not in a Pack.

H · Starter Reading List

If you have four hours this weekend.

In this specific order. Each hour builds on the last. By Sunday night you will have the patterns, the debate, the research, and the integration story in one mental model.

01
Building Effective Agents (Anthropic)
Schluntz & Zhang · Dec 2024 · 45 min
The canonical taxonomy. Start here. Then open the cookbook notebooks (orchestrator_workers.ipynb, evaluator_optimizer.ipynb) to see runnable code.
02
How We Built Our Multi-Agent Research System (Anthropic)
June 13, 2025 · 30 min
Production counterpart. Read immediately after #01. The engineering takeaways read like a checklist.
03
Don't Build Multi-Agents (Cognition)
Walden Yan · June 12, 2025 · 15 min
The counter-argument. Read this right after Anthropic's post to get the debate in proper tension.
04
How and When to Build Multi-Agent Systems (LangChain)
Harrison Chase · June 16, 2025 · 15 min
The reconciliation. By the end you'll have a position on read-heavy vs write-heavy workflows.
05
AFlow: Automating Agentic Workflow Generation
arXiv 2410.10762 · ICLR 2025 Oral · skim 30 min
Focus on Section 3 (workflow representation) and Section 4 (MCTS search). This is the academic blueprint for topology search.
06
Equipping Agents for the Real World with Agent Skills
Anthropic · Oct 16, 2025 · 20 min
Then open github.com/anthropics/skills and read skill-creator/SKILL.md. This is the reference meta-skill architecture.
07
A Field Guide to Rapidly Improving AI Products
Hamel Husain · hamel.dev · 25 min
The eval playbook. Required reading for anyone shipping agents that humans depend on.
08
The Second Half (Shunyu Yao)
April 2025 · 15 min
Why eval tooling is the next frontier, not better models. Short, important, worth re-reading.
