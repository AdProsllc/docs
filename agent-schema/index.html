100 fields across every stage of the agent lifecycle — identity, behavior, execution, economics, governance.

Ecommerce & Growth Team Adjustments

What the general schema misses for performance marketing — and the fields that turn the library into an ROI-reporting asset.

A generic agent schema treats all agents the same. Performance marketing has specific failure modes — bad attribution, premature optimization, compliance blowups, irreversible platform actions — that need to be designed into the schema, not discovered in production. These adjustments exist because a client's ad account is production infrastructure, not a sandbox.

1. Attribution context must be explicit, not implied

A "ROAS" field is meaningless without knowing whose ROAS. Every agent that reads or reports performance needs:

  • Attribution source (Meta in-platform, GA4, Triple Whale, Northbeam, Shopify native, MMM)
  • Attribution window (1d click, 7d click, etc.)
  • Blended vs. platform-specific
  • Incrementality-tested (y/n)

Without these, the Daily Client Snapshot says "ROAS is 2.8" and three stakeholders read three different numbers. This is the #1 source of agent-generated distrust.

2. Performance guardrails need dollar thresholds, not just logic

A generic "guardrails" field isn't enough. Add:

  • Max spend change per action ($ and %)
  • Min data threshold before action (spend floor, conversions floor, days floor)
  • Statistical significance required (confidence level)
  • Human approval threshold ($ above which a human signs off)
  • Pause authority (can this agent pause spend autonomously? y/n)

The Underperformer Sniper shouldn't pause a $5K/day ad set after 6 hours of data. That's a schema-level rule, not a prompt-level hope.

3. Business context fields the schema needs

Most public schemas assume agents operate on generic data. Growth teams don't:

  • Client ID / account ID
  • AOV range (changes what "good CPA" means)
  • Margin / contribution margin tier
  • Product category (supplements vs. apparel vs. SaaS — totally different benchmarks)
  • Funnel stage (prospecting / retargeting / retention)
  • Seasonality profile
  • Promo calendar awareness (is BFCM live?)
  • LTV horizon (30d / 90d / 12mo payback expectation)

An agent that flags "CPA is high" without knowing the client targets 90-day payback on a $180 AOV subscription is noise.

4. Creative-specific metadata

For creative agents, generic I/O fields aren't enough:

  • Hook type used (H.E.I.T. / HDFB / POV / problem-solution)
  • Format (UGC / static / motion graphic / founder-led)
  • Persona targeted
  • Funnel stage the creative is built for
  • Language / localization
  • Compliance flags (health claims, before/after, earnings claims, weight loss)
  • Platform-native requirements (TikTok vs. Meta vs. AppLovin have different specs)

The compliance flag alone prevents a telehealth/GLP-1 disaster.

5. Learning & benchmark fields

Growth teams compound — an agent that ran last quarter should make this quarter's agent smarter:

  • Benchmark source (account history / vertical peers / Ad Pros network)
  • Benchmark snapshot date
  • Learning feed (does this agent's output train future runs? y/n)
  • Winner repository reference (link to winning ads/angles it has produced)
  • Pattern library contribution (does output feed the 9 Dimensions library?)

6. Action-level fields (beyond "output")

Media buying agents don't just generate text — they do things. The schema needs:

  • Reversibility (can this action be undone automatically?)
  • Undo window (how long after action can auto-revert fire?)
  • Change log destination (where the edit is logged for audit)
  • Platform API permissions required
  • Dry-run mode available (y/n)
  • Diff preview shown before execution (y/n)

This is the difference between "agent updated budget" being fine vs. catastrophic.

7. Performance outcome tracking — the real one

The current schema has "success rate (% from feedback)." For growth work, feedback ≠ outcome. Feedback is "did the user like it." Outcome is "did the business metric move."

  • Outcome metric (what KPI this agent is ultimately trying to move)
  • Outcome attribution window (7d / 30d / 90d)
  • Control comparison method (holdout / pre-post / synthetic)
  • Revenue attributed (running $ total)
  • Spend saved (running $ total)
  • Hours saved (running count)
  • Decisions influenced (count)

This turns the library from a tool catalog into an ROI-reporting asset. It's also what unlocks outcome-based pricing — charge based on value delivered, not just execution count.

8. Multi-account / multi-client scoping

One field missing from the general schema: scope. Is this agent:

  • Global (runs across all clients)
  • Pod-scoped (Pod A only)
  • Client-scoped (one client)
  • Cohort-scoped (all supplement clients)

Matters for permissions, cost allocation, and benchmarking.

⚡ Top 10 highest-leverage additions for Ad Pros

If limited to 10 new fields beyond the general 100, these turn the Agent Factory from a generic LLM-ops system into something specifically defensible for performance marketing:

  1. Attribution source + window
  2. Min data threshold before action
  3. Human approval $ threshold
  4. Reversibility + undo window
  5. Outcome metric + attribution window
  6. Revenue attributed / spend saved (running totals)
  7. Client scope
  8. Compliance flags

  10. Platform-native format requirements
