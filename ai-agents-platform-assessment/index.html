Internal Engine vs. External Orchestration (Hermes + CMA)

Technical Assessment · April 2026 · Internal Distribution
Section 1

Executive Summary

AdPros has a production-grade AI Agent execution engine running in production today. The core infrastructure — atomic scheduling, encrypted credentials, idempotent execution, step-level audit trail, and auto-recovery — is complete and battle-tested. What is missing is the visibility and collaboration layer: dashboards, alerts, and team access controls. These are presentation and notification features that can be built on top of the existing foundation in 2–3 weeks without introducing external dependencies or re-architecting the execution pipeline.

This document provides a factual assessment of the current engine's capabilities, maps the specific gaps against stakeholder requirements, and compares the cost and risk profile of extending the existing system versus adopting an external orchestration platform.

Section 2

Current Engine Capabilities

The following capabilities are implemented, deployed, and actively running in production. Each item references the specific mechanism or database artifact that backs it.

▶ Execution & Reliability

  • ✓ Atomic workflow claiming via PostgreSQL SELECT...FOR UPDATE — prevents double-execution when cron overlaps or retries fire
  • ✓ Cron-based scheduler with a 50-second hard cutoff and stale run recovery (runs stuck >30 min auto-marked as failed)
  • ✓ Idempotency keys on all runs — safe against duplicate webhook deliveries and retry storms
  • ✓ Serial execution within time budget — workflows process sequentially, never exceeding the cron window
  • ✓ Configurable daily run limits per workflow (default 1, max 24) as a guardrail against runaway execution

◆ Security & Isolation

  • ✓ AES-256-GCM credential encryption at rest via WORKFLOW_ENCRYPTION_KEY — decryption is server-side only
  • ✓ Row-Level Security (RLS) on all workflow tables — strict user-level tenant isolation enforced at the database layer
  • ✓ Draft confirmation system for side-effecting operations — create, update, and delete actions require explicit user approval in chat
  • ✓ OAuth 2.0 integration flow (Slack) with HMAC-signed state tokens preventing CSRF on the callback
  • ✓ No plaintext credentials in logs, client state, or API responses — credentials referenced by encrypted record IDs only

◉ Audit Trail & Observability Data

  • ✓ workflow_runs table: status, timestamps, trigger source, credits consumed, error codes, delivery status
  • ✓ workflow_run_events table: step-level execution log — load_data → sync → compute → rank → build_report → deliver
  • ✓ workflow_brand_daily_metrics: per-brand per-run computed metrics (new ads, deactivated, hooks, offers, threat scores)
  • ✓ workflow_brand_baselines: baseline tracking for delta computation across consecutive runs
  • ✓ Full JSONB context on run_input, run_output, and provider_response for every run

⚠ Failure Handling

  • ✓ Automatic integration pause after 3 consecutive delivery failures — prevents noisy retries against broken channels
  • ✓ Stale run recovery: runs stuck >30 min are auto-marked as failed and the workflow is released for the next scheduled execution
  • ✓ Structured error categorization: insufficient_credits, integration_not_found, provider_delivery_failed, max_daily_runs_reached, and others
  • ✓ Failure counter reset on successful delivery — auto-paused integrations resume after manual re-validation

★ Cost & Usage Controls

  • ✓ Per-run credit consumption tracking with WORKFLOW_EXECUTION action type attribution
  • ✓ Usage cost events recorded per execution for billing and analytics
  • ✓ Daily run limits as guardrails (default 1, max 24) — enforced at the scheduler level, not just the UI

⚙ Extensibility & Architecture

  • ✓ Provider adapter pattern (WorkflowProviderAdapter interface) — Slack implemented, additional providers plug in via the same contract
  • ✓ Template-driven workflow creation with JSON Schema validation — no open-ended code execution by users
  • ✓ Chat-as-control-plane: users discover, configure, activate, pause, and monitor workflows from the same chat interface

Section 4

Hermes + CMA: What Are They, Really?

Before comparing build vs. buy, it is important to understand exactly what these products are, what they cost, and whether they solve the problem we actually have.

C Claude Managed Agents (CMA) — Anthropic

Launched April 8, 2026 • Public Beta • Early adopters: Notion, Rakuten, Asana

CMA is Anthropic's managed infrastructure for running Claude as an autonomous agent in sandboxed sessions. It provides secure containers, long-running sessions (minutes to hours), scoped permissions, built-in tools (bash, file ops, web search), credential vaults, and end-to-end session tracing.

Monitoring Features

  • ✓ Session dashboard with status, model, timestamps
  • ✓ Chronological event tracing per session (tool calls, reasoning, outputs)
  • ✓ Per-session and aggregate cost visibility
  • ✓ Scoped environment/permission management
  • ✓ Encrypted credential vault with audit logs

Anthropic example: 1-hour Opus session with 50K input + 15K output tokens = ~$0.71. Batch API discounts do not apply.

Critical Fit Issue

CMA is designed for autonomous agent sessions where Claude runs independently with tools for extended periods — coding tasks, research workflows, multi-step automation. AdPros AI Agents already have a purpose-built orchestration engine that handles scheduling, data sync, AI-powered analysis via Claude, and multi-channel delivery (Slack, email, WhatsApp, and more). The execution infrastructure — atomic claiming, idempotency, failure recovery, credential encryption — is already solved. CMA’s sandboxed container model and session-hour billing are designed for open-ended agent autonomy, not for adding monitoring to an engine that already runs reliably.

H Hermes Agent — Nous Research

Released February 2026 • Open Source (MIT) • 104K+ GitHub stars

Hermes is an open-source, self-improving AI agent framework. It provides an agent loop with persistent memory, multi-platform messaging (Slack, Discord, Telegram, WhatsApp, etc.), multi-agent orchestration, and a companion Mission Control Dashboard for fleet management.

Mission Control Features

  • ✓ Kanban task board and real-time activity feeds
  • ✓ Token usage dashboards with per-model cost breakdowns
  • ✓ Security posture scoring (0–100) and secret detection
  • ✓ Agent fleet management with role-based access (viewer/operator/admin)
  • ✓ Self-hosted on SQLite, zero external dependencies

Critical Fit Issue

Hermes is a conversational agent framework — it replaces the agent loop, memory system, and chat interface. AdPros already has a production chat system with its own tool-use architecture, streaming SSE, brand context resolution, and a confirmation-based draft system for side effects. Adopting Hermes would mean replacing the entire chat and agent configuration layer, not adding monitoring on top. The Mission Control dashboard is useful, but it monitors Hermes-native agents — not an independently orchestrated AI Agent system like AdPros.

What “Hermes + CMA” actually means: Hermes as the agent orchestration framework (defining behaviors, skills, memory) running on CMA as the managed execution infrastructure (sandboxing, tracing, credentials). Together they provide a full-stack agent platform — but for building new autonomous agents from scratch, not for adding visibility and collaboration to an existing production AI Agent system that already has its own orchestration, security, and audit infrastructure.

Section 5

Build vs. Buy Analysis

Now that we understand what these products actually do, here is a balanced comparison.

Build (Extend Current Engine)

Add dashboards, alerts, and collaboration on existing infrastructure

Advantages

  • + Full UX control — monitoring is part of the product, not a separate tool
  • + No vendor dependency on critical execution infrastructure
  • + Leverages existing schema and audit trail — zero data migration
  • + Single codebase — one deployment, one set of credentials
  • + No additional SaaS cost beyond existing Anthropic API usage
  • + Domain-specific features (competitive intel scoring, brand threat analysis) stay first-class

Tradeoffs

  • − Engineering time: 2–3 weeks of focused development
  • − Ongoing maintenance for dashboard and alerting features
Risk: Low

Building UI and notification hooks on top of stable, battle-tested infrastructure.

Buy (Hermes + CMA)

Replace agent execution with external framework + managed infrastructure

Advantages

  • + CMA provides session tracing and credential vaults out of the box
  • + Hermes Mission Control gives fleet management and RBAC immediately
  • + Active open-source community (Hermes 104K+ stars) and Anthropic backing (CMA)

Tradeoffs

  • − Architecture mismatch: CMA is for open-ended autonomous agent sessions; AdPros AI Agents use a purpose-built orchestration engine with multi-step execution (data sync → AI analysis → report generation → multi-channel delivery). Migrating to CMA means restructuring a working system into session-based execution, adding complexity with no functional benefit.
  • − Hermes replaces, not augments: adopting Hermes means replacing the existing chat tool-use system, streaming architecture, and draft confirmation flow — not layering monitoring on top.
  • − Ongoing cost: CMA charges $0.08/session-hour + tokens on top of Hermes hosting ($19–49/month). For 50 daily workflow runs averaging 1 min each, that is ~$2.40/day in session fees alone, plus token costs.
  • − Fragmented UX: users configure in AdPros chat, but monitoring lives in Hermes Mission Control and/or Anthropic Console — three surfaces instead of one.
  • − Data split: run history and metrics would live partly in CMA/Hermes, partly in Supabase. Unified reporting requires cross-system queries.
  • − Beta risk: CMA launched April 8, 2026 and is in public beta. Features and pricing may change.
Risk: High

These products solve for building new agent infrastructure, not adding visibility to an existing one. Requires re-platforming, not just integration.

Section 6

Recommendation

Build the visibility layer. The hard parts are done.

Hermes + CMA are impressive products — but they solve a different problem. They are built for autonomous, open-ended agent sessions. AdPros runs AI-powered agents that sync competitive data, compute threat metrics, and use Claude to generate intelligent analysis reports — delivered across Slack, email, WhatsApp, and other channels. Our execution engine already has atomic scheduling, encrypted credentials, idempotent runs, step-level audit trails, and auto-recovery. What’s missing is dashboards and notifications — a presentation layer, not an execution layer.

Adopting Hermes + CMA to get monitoring dashboards is like buying a Tesla to use the GPS — the navigation is great, but you already have a working car and just needed a phone mount.

  • ▶ Delivers all 5 of Nehal’s requirements within 2–3 weeks
  • ▶ Keeps the product experience unified — one app, one login, one place to look
  • ▶ Avoids re-platforming risk on beta-stage external infrastructure
  • ▶ Preserves full data ownership and credential security under our control
  • ▶ Ships faster than integration with Hermes + CMA would (and without the re-architecture)

Future consideration: If AdPros later expands into fully autonomous agent workloads (e.g., agents that independently research, create content, and manage ad campaigns end-to-end), CMA becomes a strong candidate for that specific use case. For the current AI Agent system, the execution engine is solved — we need visibility, not a new engine.

Section 7

Implementation Roadmap

Three focused weeks to close every gap identified in the stakeholder requirements. Each week has a clear deliverable that can be demoed independently.

Week 1
Visibility + Monitoring
Nehal: Visibility, Monitoring
  • Run history page — filterable table (status, date range, workflow name, trigger source) pulling from workflow_runs
  • Run detail view — step-by-step execution timeline from workflow_run_events with duration, status, and error context per step
  • Failure alert hooks — Slack notification on failed runs (configurable channel), using the existing provider adapter
  • Real-time status indicators — workflow health badges (healthy, degraded, paused) based on recent run outcomes
Week 2
Admin Access + Error UX
Nehal: Admin Access, Error Identification
  • Admin workflow dashboard — all users' workflows with health overview, failure trends, and management controls (built on existing /api/admin/workflows)
  • Human-readable error messages — error code to plain-language mapping for all error_summary values
  • Error detail UI — click into any failed run to see the exact step that failed, the error payload, and suggested remediation
  • Integration health indicators — last validated timestamp, consecutive failure count, and active/paused status per integration
Week 3
Collaboration + Polish
Nehal: Collaboration
  • Team-level workflow sharing — add team_id column on workflows, update RLS policies for team-scoped access
  • Role-based permissions — viewer (read-only dashboard), editor (configure workflows), admin (full management including pause/delete)
  • Workflow activity feed — audit log of who changed configuration, who activated/paused, with timestamps
  • Polish and QA — loading states, error boundaries, responsive layout, print-friendly views
