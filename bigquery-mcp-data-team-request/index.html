BigQuery MCP: Data Team Request

Connecting Noor to BigQuery via Google's official MCP server

What we're doing

Connecting our internal AI assistant (Noor) to BigQuery using Google's official BigQuery MCP server. This replaces the current ad-hoc setup, where a service account JSON sits unencrypted on the Mac Mini Noor runs on with no real query path.

The official server is fully managed by Google, runs at https://bigquery.googleapis.com/mcp, authenticates via OAuth 2.0 + IAM, and writes every query to Cloud Audit Logs. No service account keys live on our side.

Source: docs.cloud.google.com/bigquery/docs/use-bigquery-mcp

What we need from you

1. Confirm scope

  • Project ID: is ad-pros-2024 the right project for client and campaign data, or should Noor point somewhere else?
  • Datasets in scope: which datasets should Noor be able to query? (e.g. campaign performance, creative analytics, client reporting)
  • Datasets to explicitly exclude: anything with raw PII, billing data, or anything regulated that Noor should never see

2. Provision a dedicated service account

Please create a service account just for Noor so we can rotate or revoke it independently:

SA name:      noor-bq-reader@<project>.iam.gserviceaccount.com
Display name: Noor BigQuery Reader (AdPros AI assistant)

3. Grant three IAM roles, scoped tightly

PROJECT=ad-pros-2024
SA=noor-bq-reader@${PROJECT}.iam.gserviceaccount.com

# Allow MCP tool calls (project-level)
gcloud projects add-iam-policy-binding $PROJECT \
  --member="serviceAccount:${SA}" \
  --role="roles/mcp.toolUser"

# Allow running query jobs (project-level)
gcloud projects add-iam-policy-binding $PROJECT \
  --member="serviceAccount:${SA}" \
  --role="roles/bigquery.jobUser"

# Read access -- apply per dataset, NOT project-wide
# Repeat for each dataset Noor should see
bq update --source_dataset_access_file - $PROJECT:<dataset_name> <<EOF
  # add: { "role": "READER", "userByEmail": "${SA}" }
EOF

(Or grant roles/bigquery.dataViewer per dataset via the console if that's easier.)

Important

No project-wide dataViewer. We want Noor's read access scoped only to the datasets you explicitly approve.

4. Allow impersonation from one human account

So we don't have to ship a key file, grant Service Account Token Creator on the SA to a single human Google identity (probably Nehal or whoever administers Noor on our side). That person runs gcloud auth application-default login --impersonate-service-account=... once on the Mac Mini, and ADC handles token refresh from there. No key ever touches disk.

gcloud iam service-accounts add-iam-policy-binding $SA \
  --member="user:<admin-email>@adpros.com" \
  --role="roles/iam.serviceAccountTokenCreator"

5. Cost guardrails (recommended, optional)

Two reasonable limits on the SA:

  • Per-query cap: we'll set maximum_bytes_billed = 10 GB as the default in our MCP config so Noor can't accidentally trigger a multi-TB query
  • Daily quota on the SA (if you want a hard ceiling): set a custom BigQuery query usage quota in the GCP console under IAM → Quotas, scoped to this SA

What we'll do on our end

  • Wire the MCP server into Noor's agent config
  • Configure ADC with SA impersonation
  • Delete the existing ad-pros-2024-bq-sa.json from disk once the new path is verified
  • Update Noor's internal docs so she knows which datasets exist and how to query them
  • Test from both Telegram and Slack with two authorized users (TG and Nehal) before opening up to the broader team

Audit trail you'll get

Every query Noor runs will appear in Cloud Audit Logs (bigquery.googleapis.com) with:

  • Service account: noor-bq-reader
  • Query SQL
  • Bytes scanned and billed
  • Requester label (we'll inject the originating Slack or Telegram user ID as a query label so you can attribute it)
